A robotic arm that plays Flappy Bird using the predictions generated by a PPO model.

## System Architecture

```
┌─────────────────────────────────────────────────────┐
│                      gym_demo.py                     │
│                                                      │
│   FlappyBird-v0 gym                                  │
│       ↓ observations                                 │
│   latency compensation                               │
│       ↓                                              │
│   PPO model → arm tap → physical spacebar press      │
│                              ↓                       │
│               pygame KEYDOWN → env.step(flap=1)      │
└─────────────────────────────────────────────────────┘
```

---

## Hardware

- **Robot arm**: SO-101 (Feetech STS3215 servos)
- **Connection**: USB serial (`/dev/tty.usbmodem5AAF2196261`)
- **Gripper**: taps the spacebar to flap

## Usage

### Run the demo
```bash
python gym_demo.py                     # full robot
python gym_demo.py --dry-run           # no arm, simulation only
python gym_demo.py --latency-frames 3  # tune arm latency (default: 3)
```

### Arm controller (manual calibration GUI)
```bash
python controller.py
```
Sliders for all 6 joints. **Enable Torque** / **Disable Torque** buttons to toggle servo lock.

| File | Purpose |
|------|---------|
| `gym_demo.py` | runs the game, extracts states to RL model and transfers output to an action (arm tap) |
| `controller.py` | GUI joint controller for manual arm calibration |
| `FlappyBird_stage_final.zip` | PPO model trained on gymnasium |
| `human_play_and_record.py` | the user plays the game and frames are recorded as data to finetune YOLO to detect pipes and birds |
| `yolo_finetuning.ipynb` | finetunes a YOLO model based on collected data stored on Roboflow |
| `yolo.onnx` | finetuned YOLO model capabale of tracking pipes and bird |

## Observation Vector

The model uses a 12-value normalized observation:

| Index | Name | Description |
|-------|------|-------------|
| 0–2 | last_pipe | x, gap_top, gap_bottom (closest pipe) |
| 3–5 | next_pipe | x, gap_top, gap_bottom |
| 6–8 | nn_pipe | x, gap_top, gap_bottom (furthest visible) |
| 9 | bird_y | vertical position (0 = top, 1 = bottom) |
| 10 | vel_y | vertical velocity / 10 |
| 11 | rot | rotation / 90° |

## What I Tried

### Approach 1 — YOLO vision pipeline
Built a full YOLO-based vision system that detects the bird and pipes from game frames, reconstructs the observation vector, and feeds it to the PPO model. The robot arm taps based on the model's prediction.

Challenges encountered:
- **Training distribution mismatch**: The original PPO model was trained on perfect gym observations. YOLO observations differ subtly, which introduces noise as part of inputs to the model, leading to corrupted model predictions.
- **Latency from hardware and inferencing**: Both YOLO inferencing and robotics arm tapping are slightly late compared to the game running. By the time an action is executed in the game, it might not be the most optimal action as the game state might be steps ahead.

### Approach 2 — Direct gym observations
Bypasses YOLO entirely. `gym_demo.py` reads the gymnasium's internal state directly and feeds it to the RL model. Robotics arm latency is compensated by simulating the state as a future state and using the future state for the model to generate predictions.

### lerobot setup commands

```bash
lerobot-setup-motors \
    --robot.type=so101_follower \
    --robot.port=/dev/tty.usbmodem5AAF2196261

lerobot-setup-motors \
    --teleop.type=so101_leader \
    --teleop.port=/dev/tty.usbmodem5AAF2195131

    --robot.type=so101_follower \

```

### Tap configuration

| Parameter | Value | Description |
|-----------|-------|-------------|
| `HOVER_POS` | 2200 | Gripper resting position (above key) |
| `TAP_POS` | 2350 | Gripper extended position (on key) |
| `TAP_DOWN_S` | 0.08s | Time to move down to key |
| `TAP_HOLD_S` | 0.05s | Time to hold key pressed |
| `TAP_UP_S` | 0.08s | Time to return to hover |

Overlapping taps are prevented by a threading lock in `arm.tap()` — if the arm is still mid-motion, the new tap command is silently dropped.
